# コードスタイルと規約

## 一般的な規約
- **開発ガイドライン**: `AGENTS.md`に詳細な開発ガイドラインが定義されている
- **言語**: 日本語でのドキュメント・コメントを推奨（プロジェクト方針）
- **品質重視**: 動くコードだけでなく、品質・保守性・安全性を常に意識
- **ボーイスカウトルール**: コードを見つけた時よりも良い状態で残す
- **Spec駆動開発**: 仕様を明確にしてから実装
- **テスト駆動開発**: テストを書いてから実装

## コーディングスタイル
### Biome設定（`biome.json`）
- **フォーマッター**: 有効化
  - インデント: スペース2つ
  - クオート: ダブルクォート（JavaScript/TypeScript）
  - 行末: LF
- **リンター**: 有効化（推奨ルール）
- **インポート整理**: 有効化

### 適用コマンド
```bash
# フォーマット
pnpm run biome:format

# リント＆フォーマットチェック
pnpm run biome:check
```

### TypeScript
- **型ヒント**: 明示的に使用
- **strictモード**: 推奨
- **any型**: 避ける（どうしても必要な場合は `unknown` を検討）
- **意味のある変数名・関数名**: 意図を明確に
- **型エイリアス**: 複雑な型は `type` または `interface` で定義
- **null/undefined**: 明確に区別
- **オプショナルチェイニング**: 積極的に使用 (`?.`)
- **Nullish coalescing**: 積極的に使用 (`??`)

### React/Next.js
- **関数コンポーネント**: クラスコンポーネントより優先
- **Hooks**: 適切に使用（useState、useEffect、useCallback等）
- **Server Components**: デフォルトで使用（Next.js 15 App Router）
- **Client Components**: 必要な場合のみ `"use client"` ディレクティブ
- **Server Actions**: `"use server"` ディレクティブで定義
- **Props型定義**: `interface` で明示的に定義

## 命名規則
### 変数・関数
- **キャメルケース**: `getMastra`, `createx402Agent`, `weatherData`
- **Boolean値**: `is`, `has`, `should` プレフィックス（例: `isLoading`, `hasError`）
- **イベントハンドラ**: `handle` プレフィックス（例: `handleSubmit`, `handleClick`）
- **非同期関数**: 明示的に `async` キーワード使用

### クラス・型
- **パスカルケース**: `AgentCoreMastraX402Stack`, `WeatherData`, `MCPClient`
- **インターフェース**: `I` プレフィックスは使用しない（例: `WeatherData` not `IWeatherData`）

### 定数
- **アッパースネークケース**: 環境変数、設定値（例: `MCP_SERVER_URL`, `PORT`）
- **キャメルケース**: ローカル定数（例: `apiKey`, `region`）

### ファイル
- **コンポーネント**: パスカルケース（例: `Weather.tsx`, `Button.tsx`）
- **ユーティリティ**: ケバブケースまたはキャメルケース（例: `utils.ts`, `helpers.ts`）
- **設定ファイル**: ケバブケース（例: `next.config.ts`, `biome.json`）
- **インフラ**: ケバブケース（例: `cdk-stack.ts`）

## コメント規約
### 目的
- **「なぜ」を説明**: コードで表現できない理由や意図を説明
- **「何を」はコードで表現**: 自己説明的なコードを書く

### JSDoc
- **公開API**: 必須
- **関数**: パラメータ、戻り値、説明を含む
- **クラス**: 目的と使用方法を含む
- **複雑なロジック**: 必要に応じて詳細な説明

```typescript
/**

 * @param useGemini - trueの場合はGeminiモデルを使用
 * @returns Mastraインスタンス
 */
export const getMastra = async (useGemini: boolean): Promise<Mastra> => {
  // ...
};
```

### 言語
- **日本語コメント可**: プロジェクト方針に従う
- **関数名・変数名**: 英語（国際的な慣習に従う）

## エラーハンドリング
### 原則
- **`@ts-ignore` 禁止**: 型エラーは正しく修正
- **try-catch握りつぶし禁止**: エラーは適切にハンドリング
- **早期エラー検出**: 早めに失敗させる（fail-fast）
- **明確なエラーメッセージ**: ユーザーに分かりやすいメッセージ

### 実装例
```typescript
try {
  const result = await callAPI();
  return result;
} catch (error) {
  console.error("API call failed:", error);
  // エラーメッセージを詳細に表示
  const errorMessage = error instanceof Error 
    ? `${error.name}: ${error.message}` 
    : String(error);
  return { 
    text: `エラー: ${errorMessage}`,
    error: errorMessage 
  };
}
```

### 外部API・ネットワーク
- **必ず失敗を考慮**: タイムアウト、リトライ、エラーメッセージ
- **適切なステータスコード**: HTTPステータスコードを正しく使用

## DRY原則（Don't Repeat Yourself）
- **重複を避ける**: 共通ロジックは関数・モジュールに抽出
- **単一の信頼できる情報源**: 同じ情報を複数箇所に書かない
- **適切な抽象化**: 過度な抽象化は避ける

## SOLID原則
- **単一責任の原則**: 1つのクラス/関数は1つの責任のみ
- **開放閉鎖の原則**: 拡張に開いて、修正に閉じている
- **リスコフの置換原則**: 派生型は基本型と置換可能
- **インターフェース分離の原則**: クライアントに不要なインターフェースを強制しない
- **依存性逆転の原則**: 抽象に依存し、具象に依存しない

## セキュリティ
### 環境変数管理
- **APIキー・パスワード**: 環境変数で管理（`.env`、SSM Parameter Store）
- **ハードコード禁止**: ソースコードに秘密情報を含めない
- **`.gitignore`**: `.env`, `*.key`, `*.pem` 等を含める

### 入力検証
- **すべての外部入力を検証**: zodスキーマ等を使用
- **SQLインジェクション対策**: パラメータ化クエリ使用
- **XSS対策**: サニタイゼーション、エスケープ

### 権限管理
- **最小権限の原則**: 必要最小限の権限のみ付与
- **IAMロール**: 適切なポリシーを設定

## パフォーマンス
### 原則
- **推測ではなく計測**: プロファイラー、メトリクスを活用
- **初期段階から拡張性を考慮**: ただし過度な最適化は避ける
- **遅延読み込み**: 必要になるまでリソースを読み込まない
- **キャッシュ**: 有効期限と無効化戦略を明確に
- **N+1問題**: データ取得の効率化

### React/Next.js
- **useMemo/useCallback**: 不要な再レンダリングを避ける
- **React.lazy**: コード分割
- **next/image**: 画像最適化
- **next/link**: プリフェッチ

## Git運用
### コミットメッセージ
- **コンベンショナルコミット形式**:
  - `feat:` 新機能
  - `fix:` バグ修正
  - `docs:` ドキュメント
  - `test:` テスト
  - `refactor:` リファクタリング
  - `chore:` その他（ビルド、依存関係更新等）
  - `style:` コードスタイル（機能に影響なし）
  - `perf:` パフォーマンス改善
- **原子的なコミット**: 単一の変更に焦点
- **明確で説明的**: 変更内容を簡潔に説明
- **英語推奨**: 国際的なプロジェクトの慣習

### 例
```bash
git commit -m "feat: add Gemini model support to Mastra Agent"
git commit -m "fix: resolve SSM parameter loading issue"
git commit -m "docs: update README with deployment instructions"
git commit -m "refactor: simplify MCP client initialization"
```

### ブランチ戦略
- **main**: 本番環境（常に安定）
- **develop**: 開発環境（統合ブランチ）
- **feature/***: 機能開発
- **fix/***: バグ修正
- **hotfix/***: 緊急修正

## コードレビュー
### 姿勢
- **建設的な改善提案**: 批判ではなく改善
- **個人ではなくコードに焦点**: コードの品質を議論
- **変更の理由と影響を明確に**: なぜこの変更が必要か説明
- **フィードバックを学習機会として歓迎**: 成長の機会

### チェックポイント
- コードが要件を満たしているか
- テストが適切に書かれているか
- セキュリティ上の問題がないか
- パフォーマンスに影響がないか
- コードスタイルが統一されているか

## デバッグ
### ベストプラクティス
- **問題を確実に再現**: 再現手順を確立
- **二分探索**: 問題の範囲を絞り込む
- **最近の変更から調査**: git logで最近の変更を確認
- **適切なツール**: デバッガー、プロファイラー、ログ
- **調査結果と解決策を記録**: 知識を共有

### ログ
- **適切なログレベル**: debug, info, warn, error
- **構造化ログ**: JSON形式でログ出力
- **CloudWatch Logs**: AWSリソースのログは統合

## 依存関係管理
### 原則
- **本当に必要な依存関係のみ**: 不要なパッケージは追加しない
- **ロックファイル**: `pnpm-lock.yaml` を必ずコミット
- **ライセンス確認**: 新しい依存関係のライセンスを確認
- **メンテナンス状況**: アクティブにメンテナンスされているか確認
- **セキュリティパッチ**: 定期的に更新（`pnpm audit`）

### 更新
```bash
# 依存関係の確認
pnpm outdated

# セキュリティ監査
pnpm audit

# 依存関係の更新
pnpm update
```
